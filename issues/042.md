## Fix Remaining Broad Exception Handlers in main.py

After the 4 worst offenders are fixed (Issue 014-A), apply the same approach to the remaining ~35 broad `except Exception` handlers in this file.

## Scope
- **File**: `main.py`
- **Count**: ~35 remaining handlers (total 38, minus 3 worst offenders)
- **Depends on**: Issue 014-A (to establish pattern)

## Categorization Rules (from docs/16-logging-system.md)

Examine each handler's context and categorize:

**ERROR level** — Critical failures that indicate bugs or serious issues:
- Socket/network errors
- File I/O errors
- Assertion failures
- Unexpected state errors

**WARNING level** — Expected fallback paths:
- Timeouts or retries
- Graceful degradation
- "Falling back to" patterns

**DEBUG level** — Tolerant/polling operations:
- UI automation tree walks
- Window/process scans
- Periodic polling with expected failures
- Frame processing in loops

## Implementation Approach
1. Add `import logging` and `log = logging.getLogger(__name__)` at module top if not present
2. For each `except Exception` handler:
   - Review the try block to understand what operation it guards
   - Categorize as ERROR, WARNING, or DEBUG using rules above
   - Replace `pass` with `log.LEVEL("operation context", exc_info=True)`
   - If handler has other statements besides pass (e.g., `continue`, `return None`), keep them after the log call
3. Run linter to ensure consistency

- [ ] All 35 remaining broad exception handlers have logging added
- [ ] Handlers categorized correctly (ERROR for critical, DEBUG for tolerant)

- [ ] Control flow preserved (pass stays pass, continue stays continue, etc.)
- [ ] File imports logging and has `log = logging.getLogger(__name__)`
- [ ] No silent failures remain in critical paths
