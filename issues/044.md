## Fix Broad Exception Handlers in Remaining Files

After the 4 worst offenders are fixed (Issue 014-A), apply the same approach to broad `except Exception` handlers in auxiliary files.

## Scope
- **Files**: 
  - `probe_uia.py` (~7 handlers)
  - `test_permission_scan.py` (~6 handlers)
  - `cyrus_hook.py` (~2 handlers)
- **Total**: ~15 handlers
- **Depends on**: Issue 014-A (to establish pattern)

## Special Considerations

### cyrus_hook.py
Per `docs/16-logging-system.md`: This file must stay silent (no output). Do not add logging here as it may block Claude Code if stderr is captured. Confirm with maintainer before adding any output.

### probe_uia.py & test_permission_scan.py
These are diagnostic tools. Keep `print()` calls as-is. The `except Exception` handlers here can remain silent since they're for manual inspection, not production.

However, if converting to use logging for consistency:
- Use `logging.basicConfig()` for simple setup
- Log at DEBUG level to keep output clean during normal use

## Implementation Approach
1. For **probe_uia.py** & **test_permission_scan.py**:
   - Add `import logging` and basic setup
   - Replace silent `except Exception: pass` with `log.debug("operation context", exc_info=True)`
   - Or keep as print() if preferred for diagnostic tools

2. For **cyrus_hook.py**:
   - DO NOT ADD OUTPUT — confirm no logging should be added

## Acceptance Criteria
- [ ] All 7-8 handlers (probe_uia.py & test_permission_scan.py) have logging or documented reason to keep silent
- [ ] cyrus_hook.py handlers left unchanged (or explicitly approved for logging)
- [ ] Control flow preserved
- [ ] All handlers use `exc_info=True` if logging
