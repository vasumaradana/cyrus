## Design Decisions

### D1. Logger name: `__name__` vs `"cyrus.voice"`

The issue says `log = logging.getLogger(__name__)`. The logging spec (docs/16) says `"cyrus.voice"`. Since `cyrus_voice.py` is a top-level script, `__name__` resolves to `"__main__"` when run directly or `"cyrus_voice"` when imported. Following plan 040's precedent — use `__name__` as the issue specifies. When `cyrus_log.py` lands, these can be updated to `"cyrus.voice"`.

### D2. Log level rationale

Per docs/16 and the issue categorization rules:
- **DEBUG** (10 handlers): Optional features (chimes), cleanup/shutdown, queue draining, frame processing, temp file removal — all tolerant/polling operations where failure is expected or cosmetic.
- **WARNING** (4 handlers): Network send (message lost but reconnect handles it), Kokoro load (falls back to Edge TTS), clipboard read (user action degrades gracefully), brain connection (retry pattern).
- **ERROR** (3 handlers): TTS worker (assistant can't speak), brain reader (protocol/socket error), voice loop session (critical disconnect).

### D3. Control flow preservation

- 6 handlers with `pass` only: `pass` removed, `log.LEVEL(...)` becomes the body.
- 1 handler with `break`: `log.debug(...)` inserted before `break`.
- 1 handler with `continue`: `log.debug(...)` inserted before `continue`.
- 4 handlers with `print(...)`: `print(...)` replaced with `log.LEVEL(...)`.
- 5 handlers with `print(...)` + flow control: `print(...)` replaced with `log.LEVEL(...)`, flow control preserved.

Net effect: every handler's runtime control flow is unchanged.

### D4. No exception narrowing

After analysis, all 17 handlers guard operations that can raise diverse exception types:
- Audio: `pygame`, `sounddevice`, `numpy` → various `RuntimeError`, `OSError`, `ValueError`
- Network: `asyncio` streams → `ConnectionError`, `OSError`, `BrokenPipeError`
- Model loading: `onnxruntime`, `kokoro_onnx` → import errors, runtime errors
- Cleanup: `os.unlink`, `pygame.mixer`, `keyboard` → `OSError`, `RuntimeError`

Broad catch is correct for all. All handlers keep `except Exception:` and add logging.

### D5. Handlers with existing `print(f"...{e}")` → `log.LEVEL(..., exc_info=True)`

Five handlers already print the exception message. Replacing with `log.LEVEL("context", exc_info=True)` is strictly better: it preserves the full traceback (not just `str(e)`), adds proper log levels, and integrates with the logging framework. The `exc_info=True` makes the separate `{e}` interpolation redundant.
