# Implementation Plan: Fix repeated verification failure — Command not found: cyrus2/.venv/bin/python

## Gap Analysis
**Already exists**: `.barfrc` with BUILD_COMMAND, CHECK_COMMAND, TEST_COMMAND configured. Commands work correctly when run from the cyrus directory. All 761 tests pass. Ruff check and format pass.

**Needs building**: The `.barfrc` commands use relative venv paths (`cyrus2/.venv/bin/python`) as the binary. Barf's `preCheckCommand` validates the binary exists before running the command. When `cwd` is not threaded through correctly (or `process.cwd()` differs from the project root), `accessSync('cyrus2/.venv/bin/python')` fails with ENOENT because the relative path resolves against the wrong directory.

## Root Cause Analysis

The barf-ts verification pipeline has a `preCheckCommand()` function (`barf-ts/packages/core/src/core/verification/checks.ts:78-115`) that validates command binaries before execution. For commands with `/` in the first token:
- If `cwd` is provided: resolves `${cwd}/${firstToken}` and checks with `accessSync`
- If `cwd` is NOT provided: checks the raw relative path against `process.cwd()`

The error `Command not found: cyrus2/.venv/bin/python (resolved to cyrus2/.venv/bin/python)` proves `cwd` was falsy during pre-check. While barf-ts code threads `projectCwd` through most paths, the commands can be made robust by avoiding relative paths with `/` in the binary token.

**Two-part fix**:
1. **Cyrus fix (this issue)**: Change `.barfrc` to use `uv run` which puts `uv` (no `/`) as the first token — pre-check uses `which uv` instead of `accessSync`
2. **Barf-ts fix (separate issue)**: Ensure `cwd` is always threaded — not in scope here

## Approach

Replace `cyrus2/.venv/bin/python -m <tool>` with `uv run --project cyrus2 <tool>` in all three commands. This is the right approach because:
- `uv` is installed and in PATH (`/home/daniel/.platformio/.cache/uv`)
- The cyrus2 project already uses uv (`uv.lock` and `pyproject.toml` exist)
- `uv run` automatically uses the project's venv and dependencies
- The first token `uv` has no `/`, so pre-check uses `which` (more reliable)
- All three commands verified working with `uv run`
