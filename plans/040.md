## Design Decisions

### D1. Logger name: `__name__` vs hardcoded

The issue says `log = logging.getLogger(__name__)`. The logging spec (docs/16) says `logging.getLogger("cyrus.brain")`. Since these files are top-level scripts (not inside a package), `__name__` resolves to `"__main__"` when run directly or `"cyrus_brain"` / `"main"` when imported. The issue explicitly requests `__name__`, so we follow the issue. When issue 009 lands and configures the `"cyrus"` root logger, these can be updated to `"cyrus.brain"` / `"cyrus.main"` if needed.

### D2. Log level rationale

Per docs/16 level mapping:
- **UIA tree walks** → `DEBUG`: Polling operations that fail routinely when UI elements disappear mid-walk. High frequency, expected failures.
- **Window tracking** → `DEBUG`: Polling loop, race condition with window focus. Expected failures.
- **Session scan (both handlers)** → `ERROR`: Critical operation — failure means sessions aren't detected. Infrequent, signals real problems.
- **VAD loop** → `DEBUG`: Frame-level processing, high frequency. A single bad frame is not actionable.

### D3. Control flow preservation

The log call replaces `pass` in handlers where `pass` was the only body statement. This is idiomatic Python — `pass` exists only to satisfy the grammar when a block has no other statements. Once `log.debug(...)` is the body, `pass` is redundant and removed.

For the VAD handler, `continue` is preserved explicitly because it has semantic meaning — it skips the rest of the loop iteration. The log call is inserted _before_ `continue`.

Net effect: every handler's runtime control flow is unchanged.

### D4. No exception narrowing

The issue's approach section says "If specific exception types can be caught safely, use them". After analysis:
- UIA handlers: `uiautomation` raises various COM/Windows errors (`comtypes.COMError`, `LookupError`, `OSError`) unpredictably. Broad catch is correct for resilience.
- Window tracking: `pygetwindow` can raise `AttributeError`, `IndexError`, or Windows-specific errors. Broad catch is correct.
- Session scan: Same UIA + window enumeration errors. Broad catch is correct.
- VAD: `numpy`/`torch` can raise various errors on malformed audio frames. Broad catch is correct.

All handlers keep `except Exception:` and add logging. This matches the issue's fallback approach.
