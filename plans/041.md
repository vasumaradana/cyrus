## Design Decisions

### D1. Logger placement

`import logging` goes in the stdlib imports block (after `import socket`, line 33). `log = logging.getLogger(__name__)` goes after all imports, before the `# ── Configuration ──` section (after `from collections import deque`, line 60). This matches issue 040's plan for consistency.

For the two import-level handlers (lines 42, 56) that execute before `log` is defined, use the module-level `logging.warning()` / `logging.error()` functions directly instead of `log.warning()`.

### D2. Logger name: `__name__`

Per issue specification. Resolves to `"__main__"` when run directly or `"cyrus_brain"` when imported. Consistent with issue 040.

### D3. Level categorization rationale

Per `docs/16-logging-system.md`:
- **ERROR**: Socket/network errors, file I/O errors, unexpected state errors — critical operational failures
- **WARNING**: Timeouts, retries, graceful degradation, "falling back to" patterns
- **DEBUG**: UIA tree walks, window/process scans, periodic polling with expected failures, frame processing

### D4. No exception narrowing

UIA, pygetwindow, pyautogui, comtypes, and pyperclip raise unpredictable exception types (`COMError`, `LookupError`, `OSError`, `AttributeError`, Windows-specific errors). Broad `except Exception:` is correct for resilience. We add logging, not exception type changes.

### D5. Control flow preservation

Every handler keeps its original flow: `pass` stays `pass` (or is replaced by the log call as the new body), `continue` stays `continue`, `break` stays `break`, `return` stays `return`, reconnect loops stay intact. Log call is inserted *before* flow control statements.

### D6. Print-to-log conversion

9 handlers already have `print()` output. These are converted to the appropriate `log.LEVEL()` call with `exc_info=True`. The `print()` is removed (logging replaces it, not supplements it).
