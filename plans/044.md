## Gap Analysis

**Already exists**:
- All three target files exist in both `cyrus/` (legacy) and `cyrus/cyrus2/` (active):
  - `probe_uia.py` — 7 broad `except Exception` handlers (all `pass` except one `print()` fallback)
  - `test_permission_scan.py` — 6 broad `except Exception` handlers (mix of `pass` and `print()`)
  - `cyrus_hook.py` — 2 broad `except Exception` handlers (one `pass`, one `sys.exit(0)`)
- Pattern established by issue 040: `import logging; log = logging.getLogger(__name__); except Exception: log.debug("context", exc_info=True)`
- `docs/16-logging-system.md` explicitly notes: `probe_uia.py` = diagnostic tool (keep `print()`), `cyrus_hook.py` = MUST STAY SILENT (no logging)
- Test infrastructure at `cyrus2/tests/` with pytest and ruff configured

**Needs building**:
- Add `import logging` + logger setup to `probe_uia.py` and `test_permission_scan.py` (both versions)
- Replace silent `except Exception: pass` with `log.debug("context", exc_info=True)` in those files
- Leave `cyrus_hook.py` unchanged with documented justification
- Verify ruff passes on all modified files

## Approach

**For probe_uia.py & test_permission_scan.py**: Add `import logging` and `log = logging.getLogger(__name__)`. Replace each silent `except Exception: pass` with `log.debug("<descriptive context>", exc_info=True)`. Keep existing `print()` statements as-is (these are diagnostic tools meant for manual inspection). Preserve all control flow exactly.

**For cyrus_hook.py**: Leave all handlers UNCHANGED. Per `docs/16-logging-system.md`, this file must stay completely silent — no logging, no output — because it's invoked by Claude Code and any stderr output could block the extension. Document this explicitly in the acceptance criteria.

**Why this approach**: The issue itself says diagnostic tools can keep `print()`. Adding `log.debug()` with `exc_info=True` to silent handlers provides debuggability without changing user-visible behavior, since DEBUG messages won't appear unless explicitly enabled.

**Both versions**: Changes must be applied to both `cyrus/` (legacy) and `cyrus/cyrus2/` (active) since both exist and are nearly identical.
