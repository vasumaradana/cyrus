# Plan 044: Replace broad exception handlers — probe_uia, test_permission_scan, cyrus_hook

## Summary

Address all 15 `except Exception` handlers across 3 auxiliary files: add `log.debug()` to silent handlers in the two diagnostic tools (`probe_uia.py`, `test_permission_scan.py`) and document why `cyrus_hook.py` must stay completely silent. Follows the pattern established by plans 040–043.

## Design Decisions

### D1. Diagnostic tools get `log.debug()`, not `print()`

Per the issue: "if converting to use logging for consistency, log at DEBUG level to keep output clean during normal use." The silent `except Exception: pass` handlers in both diagnostic tools get `log.debug("context", exc_info=True)`. Since `logging` defaults to WARNING level without explicit configuration, these messages are suppressed during normal diagnostic runs and only surface when `CYRUS_LOG_LEVEL=DEBUG` is set. This preserves the current behavior (silent) while adding traceability for deeper debugging.

### D2. Print-based handlers stay as print

Four handlers already communicate errors via `print()`:
- `probe_uia.py:103` — prints `"(unreadable)"` as part of diagnostic table output
- `test_permission_scan.py:29` — prints `"<error: {e}>"` as part of UIA tree dump
- `test_permission_scan.py:88` — prints `"<no rect>"` as part of Chrome pane listing
- `test_permission_scan.py:110` — prints `"Error: {e}"` as top-level error display

These are intentional user-facing output in diagnostic tools, not silent failures. Converting to logging would break the diagnostic output format. Documented as "already communicating" per the AC: "have logging or documented reason to keep silent."

### D3. cyrus_hook.py must stay silent — no logging

Per `docs/16-logging-system.md`: "cyrus_hook.py intentionally has no output. Adding logging there risks blocking Claude Code if stderr is captured." Both handlers are correct as-is:
- `_send()` (line 23): Brain not running is the expected state when Claude Code starts before the brain. Silent failure is the correct behavior — a crashing or blocking hook would halt Claude Code.
- `main()` (line 30): Invalid JSON on stdin means the hook was invoked incorrectly. Clean `sys.exit(0)` is correct — any output could be captured and interpreted as hook response data.

### D4. No `logging.basicConfig()` in diagnostic tools

The issue suggests `logging.basicConfig()` for simple setup. However, adding `basicConfig()` in a module risks interfering with the root logger if these files are ever imported alongside `cyrus_log.py`. Instead, we just add `import logging` + `log = logging.getLogger(__name__)`. Python 3.2+'s "last resort" handler shows WARNING+ to stderr by default, and DEBUG messages are suppressed — which is exactly what we want.

### D5. No exception narrowing

Same rationale as plans 040–043: `uiautomation` raises diverse COM/Windows errors (`comtypes.COMError`, `LookupError`, `OSError`) unpredictably. Broad `except Exception:` is correct for resilience in UIA tree walks.

### D6. Control flow preservation

- Silent `pass` handlers: `pass` removed, `log.debug(...)` becomes the body
- Print-based handlers: unchanged
- `sys.exit(0)` handler: unchanged
- No `continue`, `break`, or `return` statements affected in any handler
