## Root Cause Analysis

Barf's `preCheckCommand()` (in `barf-ts/packages/core/src/core/verification/checks.ts`) extracts the first token from shell commands. When the token contains `/` (like `cyrus2/.venv/bin/python`), it tries to resolve it as a filesystem path using `accessSync(resolvedPath, X_OK)`. When `cwd` is undefined or falsy, the relative path is used directly, and `accessSync` fails because Node's process cwd may differ from the project directory.

The fix: restructure commands so the first token is a PATH-resolvable command name (no `/`), which uses the `which`-based check path instead.

## Approach

Change `.barfrc` to use `ruff` directly (globally available in PATH) for build/check, and use `env` to prepend the venv to PATH for test (since pytest is only in the venv). This avoids relative paths in the first token, bypassing the `accessSync` pre-check entirely.

**Why this approach over alternatives:**
- Using absolute paths: not portable, breaks if project moves
- Installing pytest globally: changes system state, not reproducible
- Fixing barf-ts preCheckCommand: out of scope (different repo), and projects should use robust command formats anyway
