## Gap Analysis
**Already exists**: The venv at `cyrus2/.venv/bin/python` exists and works. All tools (ruff, pytest) are installed. Commands work when run manually from the project root.
**Needs building**: The `.barfrc` commands use relative venv paths (`cyrus2/.venv/bin/python`) that fail barf's `preCheckCommand` validation. The precheck extracts the first token, sees it contains `/`, tries `accessSync(relativePath, X_OK)` ‚Äî this fails when barf's `cwd` parameter is undefined or doesn't match the project root.

## Root Cause

In `barf-ts/packages/core/src/core/verification/checks.ts`, `preCheckCommand()` extracts the first token from the shell command string. When the token contains `/`, it treats it as a file path and calls `accessSync(resolvedPath, X_OK)`. When `cwd` is undefined, `resolvedPath = firstToken` (the raw relative path), and `accessSync` resolves it relative to the barf **process** working directory ‚Äî not the project directory. This causes:

```
Command not found: cyrus2/.venv/bin/python (resolved to cyrus2/.venv/bin/python)
```

The error message itself proves `cwd` is undefined: if cwd were set, the resolved path would be prefixed with the absolute project path.

## Approach

**Change `.barfrc` to use `uv run` instead of relative venv paths.** This is the correct fix because:

1. `uv` is a PATH-resolvable binary ‚Äî `preCheckCommand` will use `which uv` instead of `accessSync`, which works regardless of working directory
2. `uv run --project cyrus2` handles venv activation transparently
3. All three commands have been verified to work with `uv run`:
   - `uv run --project cyrus2 ruff check cyrus2/` ‚Üí ‚úÖ passes
   - `uv run --project cyrus2 ruff format --check cyrus2/` ‚Üí ‚úÖ passes
   - `uv run --project cyrus2 pytest cyrus2/tests/` ‚Üí ‚úÖ passes (761 tests)
4. This is more robust than fixing barf-ts code paths, and follows the `uv` workflow the project already uses

**Why not fix barf-ts instead?** While there may be a barf-ts bug where `projectCwd` isn't propagated in some code paths, fixing the `.barfrc` is the correct project-level fix: commands should use PATH-resolvable tools, not fragile relative paths to venv internals.
